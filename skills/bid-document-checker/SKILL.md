# 投标文件检查技能

当前版本：v3.0.6
更新日期：2026-07-10
本版定位：薄入口 + 一套验收卡体系 + 轻量检查摘要/正式报告双输出模式。检查体系统一为 `PRE 前置检查 + G01-G06 通用检查 + D00-D11 内容域检查`，避免双轴重复检查。

本技能的验收卡体系是投标文件正式验收标准的唯一来源；其他岗位自检卡和主控运行门禁卡不得替代本技能的检查标准。结构化中间文件（blueprint、blocks、字段 JSON、模板路径）由 `bid-reviewer` 做过程复核；本技能正式检查最终 DOCX/PDF/交付稿结果。过程检查可使用 `lightweight_check` 只返回结构化摘要；正式检查使用 `formal_report` 生成本轮整合验收卡、Markdown 报告和 HTML 用户版。

## 1. 适用范围

当用户要求检查投标文件、标书、章节稿、初稿、定稿、盖章版或终检 PDF 时使用本技能。

支持四类阶段：

| 阶段 | 典型输入 | 核心目标 |
|---|---|---|
| 章节版本 | 单个章节、局部 MD、方案小节 | 判断本章节是否可进入合并或继续写作 |
| 初稿版本 | 全书初步合并，资料/报价/签章可能未齐 | 找出结构、响应、资料和口径缺口 |
| 定稿版本 | 内容、报价、资料基本齐，准备签章 | 判断是否可进入签章 |
| 盖章版本 | 已签字盖章 PDF 或提交前文件 | 判断是否建议提交 |

若用户已说明阶段，直接采用。若无法从文件名、路径和内容高置信判断阶段，先询问用户；不要猜。

## 2. 必读输入

检查前按实际可用性读取：

1. 待检查投标文件：优先 PDF；非 PDF 需说明页码、签章、图片核验限制。
2. 招标文件、补遗答疑、招标解析产物（有招标文件时）。
3. 项目已确认口径（如有）：响应矩阵、资料清单、整改闭环、关键口径。
4. 上游/辅助成果（如有）：固定格式摘录、证明材料挂接表、DOCX/PDF 交接说明、`bid-reviewer` 结构化过程复核结论、`document-recognizer` 识别 JSON。
5. 本技能 `references/acceptance-card.md`。

项目级验收指南仅在用户指定、项目有特殊口径或需要解释规则来源时读取；默认以本技能验收卡执行。

优先级：招标文件/补遗答疑 > 项目已确认口径 > 招标解析产物 > 本技能验收卡 > 通用经验。

## 3. 入口模式与关联 Skill

| 模式 | 条件 | 处理 |
|---|---|---|
| `project_mode` | 有项目控制台/响应矩阵 | 读取项目口径完整检查 |
| `ad_hoc_mode` | 只有招标文件 + 投标文件 | 无解析产物先用 `tender-document-parser`；临时产物放 `<投标文件目录>/.bid-checker/`；不强找项目控制台 |
| `bid_only_limited_mode` | 只有投标文件 | 只查内部一致性、报价勾稽、签章页码、图片/排版；招标响应结论写受限 |

招标解析质量门未过，不得无保留通过。盖章版 PDF 且模型无视觉能力时，使用 `document-recognizer`；仍无法核验则标 `受限未执行`。
招标解析产物按 `requirements`、`fatal_checklist`、`scoring_matrix`、`material_checklist`、`fixed_formats`、`timeline_matrix` 映射到本轮检查项；大纲、目录、分册、投标文件格式和章节结构要求优先映射到 `D00`；不得替代招标原文。

## 4. 输出模式

检查前必须确定输出模式：

| 输出模式 | 适用场景 | 输出要求 |
|---|---|---|
| `lightweight_check` | 大纲复核、资料复核、写作门禁、章节过程复核、主控/解析审查内部调用，或用户未明确要求报告的过程检查 | 使用本验收卡标准判断，只输出结构化摘要，不生成 Markdown/HTML/本轮验收卡文件 |
| `formal_report` | 初稿整本检查、DOCX 成稿检查、定稿/盖章版/提交前终检、用户明确要求“生成报告/输出文件/专项报告”、需要对外或跨团队流转的正式检查 | 生成本轮整合验收卡文件、Markdown 报告和 HTML 用户版；对用户交付时优先提供 HTML |

`lightweight_check` 不是降低标准，只是不落盘报告。摘要必须包含：结论、检查阶段、检查项覆盖摘要、P0/P1/P2、依据、最小返工路径、受限项和是否需用户确认。

结构化生产边界：本技能不逐个审查 `document_blueprint.json`、`*_blocks.json` 或字段 JSON 的中间内容；这些由 `bid-reviewer` 过程复核。本技能只看最终文件结果是否正确渲染，若最终 DOCX/PDF 残留 `{{...}}` 占位符、缺图、漏表、空表、固定格式未渲染或内部块外露，按正式验收问题处理。

## 5. 执行流程

必须按以下顺序执行：

1. 判断入口模式。
2. 判断输出模式：默认过程检查用 `lightweight_check`；正式投标文件检查或用户明确要求报告时用 `formal_report`。
3. 必要时先完成招标解析或文档识别。
4. 确认检查阶段、检查轮次、检查对象和版本。
5. 形成本轮检查项：以 `PRE + G01-G06 + D00-D11` 为骨架，先判断是否存在招标强制大纲/目录/分册/格式要求；无强制大纲时使用自拟大纲标准，再映射招标解析产物和项目口径。
6. `formal_report` 需要将本轮检查项落盘为本轮整合验收卡；`lightweight_check` 只在回报中列出覆盖摘要，不生成验收卡文件。
7. 检查映射覆盖率：未映射要求必须进入 `TENDER-xx`、人工复核或受限说明；未处理不得输出完整通过。
8. 按本轮检查项逐项检查并回填结果，不得批量写“均通过”。
9. 先输出 P0/P1/P2 当前问题清单，作为唯一问题台账；每个真实问题只列一行，绑定全部关联检查 ID、证据位置和整改建议。
10. G/D/PRE 明细的 `检查范围` 只写核对对象和动作，不写页码证据、问题编号、风险或建议，原则上不超过 120 字。`结果与证据` 中通过项只写“通过”或一处最小定位；问题项只写最小证据定位和关联问题编号，不重复完整问题、风险和建议。
11. `lightweight_check` 输出结构化摘要并停止，不生成 Markdown、HTML、验收卡文件或预览链接。
12. `formal_report` 先输出 Markdown 结构源，再按 `references/report-contract.md` 生成同内容 HTML 用户版；向用户回报时必须优先提供 HTML 文件路径或预览入口，Markdown 作为归档和审计补充。
13. `formal_report` 完成后询问用户是否需要生成微信可打开的临时 HTML 预览链接；未获确认不得自动发布公网链接。
14. 需要 PDF 报告时，再询问用户是否导出 PDF。

## 6. 验收卡体系

本技能只使用一套验收卡，结构见 `references/acceptance-card.md`。

验收卡分两层：

- 全局模板：`references/acceptance-card.md`，只定义通用骨架和基础检查项。
- 本轮检查项：每次检查都必须先形成。`formal_report` 落盘为本轮整合验收卡文件，项目模式放在检查输出目录，临时模式放在 `.bid-checker/`；`lightweight_check` 只在结构化摘要中列出覆盖情况，不落盘验收卡文件。

最小分组：

- `PRE`：前置检查。
- `G01-G06`：通用检查，轻量覆盖横向底线。
- `D00-D11`：内容域检查，覆盖投标大纲和标书核心模块。

通用检查不重复展开内容域细项；内容域问题只写一次，可同时在报告中按通用标签汇总。

本轮检查项生成规则：

- 每条有效招标要求至少映射到一个主责检查项。
- 能归入 `G/D` 的直接归入；找不到合适位置时才新增 `TENDER-xx`。
- 一个问题只编号一次；P0/P1/P2 数量只按 `当前问题` 清单中的唯一编号统计，关联检查项引用同一问题编号。
- `formal_report` 的实例卡必须记录路径、总检查项数、项目增量项数、未映射要求数和 `TENDER-xx` 数量；`lightweight_check` 必须在摘要中写明这些数量或受限原因。

## 7. 风险分级

| 等级 | 判定范围 | 处理口径 |
|---|---|---|
| P0 | 废标、资格不满足、报价红线、固定格式破坏、盖章版本签章缺失、实质性负偏离、最终提交阻断 | 置顶，未清零不得建议提交 |
| P1 | 重大扣分、关键证明缺失、核心口径冲突、材料过期、证据不可追溯 | 必须整改或由主控确认风险接受 |
| P2 | 格式、可读性、编号、中文标点、轻微表述优化 | 建议整改，可由主控决定是否接受 |
| NA | 不适用或未发现问题 | 写明原因或证据 |

不得输出“综合评分”“预计得分”“可得 XX 分”等模拟评委打分。

## 8. 视觉核验模式

检查结果必须写明视觉核验方式；`formal_report` 写入报告，`lightweight_check` 写入摘要：

| 模式 | 使用条件 | 结论口径 |
|---|---|---|
| `model_vision` | 当前模型可直接查看 PDF 页面截图或扫描件 | 可做图像级核验 |
| `recognizer_tool` | 调用 `document-recognizer` 或等效工具 | 消费结构化识别结果后由本技能判定 |
| `hybrid` | 模型视觉和识别工具均使用 | 冲突项标为疑似并人工复核 |
| `text_only_limited` | 无视觉能力且无识别工具 | 签章、扫描件、图片证明只能写受限，盖章版本不得无保留通过 |

`document-recognizer` 只输出可见事实；本技能负责合规解释。CA/电子签章有效性、证书/信用截图外部真伪、递交平台状态属于边界外事项，不计入受限未执行。

## 9. 报告输出

本节仅适用于 `formal_report`。`lightweight_check` 不生成 Markdown、HTML、验收卡文件或预览链接。

`formal_report` 报告结构以 `references/report-contract.md` 为准。

`formal_report` 必须同时输出：

- HTML 用户版：用户默认阅读入口，最终回报必须优先提供该文件路径或用户确认后的预览入口。
- Markdown 报告：内容结构源和审计归档，优先复制 `references/md-report-template.md` 填写。

HTML 不得删减 Markdown 的主要检查内容，不得新增与 Markdown 不一致的结论，不得重新编排 Markdown 章节顺序或生成独立报告结构。明细表允许把“检查 ID + 检查项 + 检查范围”合并为一个视觉列，并把长证据定位折叠展示；信息仍必须完整存在。

## 10. 微信临时预览链接（可选）

默认不生成公网链接。只有当用户明确要求“微信预览链接”“临时链接”“发给手机看”或在报告完成后确认需要时，才运行：

```bash
python3 scripts/preview_html.py --html <报告.html>
```

执行规则：

- 只发布单个 HTML 文件的临时副本，不得暴露项目目录、skill 目录、原始标书目录或资料目录。
- 脚本会将 HTML 复制到系统临时目录下的 `bid-html-preview/.../index.html`，再通过 Cloudflare Tunnel 暴露该临时目录。
- 输出链接后必须提醒：链接在本机服务和 Cloudflare Tunnel 运行期间有效，知道链接的人都能访问该 HTML。
- 用户要求关闭时运行：

```bash
python3 scripts/preview_html.py --stop
```

此功能只解决临时预览，不替代正式交付、权限控制、对象存储临时签名链接或 PDF 报告。

## 11. 校验器

`formal_report` 完成报告后，优先运行：

```bash
python3 scripts/validate_report.py --md <报告.md> --html <报告.html>
```

校验器用于检查：

- 是否包含 `PRE + G01-G06 + D00-D11`。
- P0/P1/P2 编号和数量是否一致。
- HTML 是否遗漏 Markdown 中的主要章节和问题编号。
- 检查范围是否塞入页码证据、问题编号或整改建议。
- 是否出现空 P0 标题、多余首页摘要、模拟评分、图片核验字段不完整等问题。

## 13. 铁律

1. 招标文件和已确认项目口径是最高依据。
2. 本轮检查项必须逐项回填，未回填不得输出完整结论。
3. 每个问题必须有检查 ID、证据位置、P0/P1/P2 状态和整改建议，并只完整写入 `当前问题` 清单；G/D/PRE 明细的检查范围、证据和问题编号必须分列归口，不得互相复制。
4. 不能因为通用检查项反向创造招标未要求的材料。
5. 固定格式原文只读，能填的只填留空、下划线或招标明确要求填写的位置。
6. 盖章版本必须完成图像级核验；无法完成时结论必须写受限。
7. `formal_report` 报告完整性优先于页面简洁性；HTML 可以折叠，不能删内容。
8. 临时预览链接必须用户确认后才生成，且只能暴露单个 HTML 副本。
9. 未映射的资格项、废标项、评分项、证明材料项、固定格式项或时间周期项未处理时，不得输出无保留通过。
